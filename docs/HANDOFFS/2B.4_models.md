# Milestone

2B.4 — database/models.py

# Summary

Implemented the Version 1 data models as plain Python dataclasses, using
only the standard library (`dataclasses`, `typing.Optional`). `database/models.py`
defines exactly five public classes, one per schema table, mirroring
`database/schema.sql` field-for-field:

- `Source` — `name`, `id`.
- `Trader` — `source_id`, `name`, `id`, `external_trader_id`, `created_at`.
- `RawMessage` — `source_id`, `raw_text`, `content_hash`, `id`, `external_id`,
  `metadata`, `received_at`, `ingested_at`.
- `TradeSignal` — `raw_message_id`, `trader_id`, `symbol`, `action`, `id`,
  `option_type`, `price`, `expiration`, `position_size`, `created_at`,
  `updated_at`.
- `TradeSignalEdit` — `trade_signal_id`, `previous_values`, `id`, `edited_at`.

Required (non-default) fields on each dataclass match the schema's `NOT NULL`
columns; all other fields are `Optional[...] = None`, including `id` (unset
until a row is persisted) and DB-defaulted timestamp columns (`created_at`,
`updated_at`, `ingested_at`, `edited_at`), which are left `None` so the
database's `DEFAULT CURRENT_TIMESTAMP` can apply. Per ChatGPT's design
review, `TradeSignal.price` is `Optional[str]` (a normalized decimal
string), not `Optional[Decimal]` — the model mirrors the column's stored
type exactly, and any `Decimal` handling is deferred to whatever code reads
or writes that field later (`repository.py` / `service.py`).

No methods beyond those generated by `@dataclass` were added: no
`__post_init__`, no validation logic, no business logic, no SQL, no
serialization/deserialization helpers, no database access.
