# Milestone

2B.6b — Edit-on-Update Rule (`database/service.py`)

# Objective

Ensure `TradeService` never updates a `trade_signals` row without first
writing a full-row JSON snapshot of the pre-edit values into
`trade_signal_edits`, per Section 6 of `docs/DATABASE_DESIGN_V1.md` and the
roadmap's acceptance criterion for 2B.6b: no code path in `TradeService` can
update a signal without a corresponding edit record.

Per Product Owner direction during design review, validation of the update
fields must have a single source of truth shared between `TradeService` and
`database/repository.py`, rather than being duplicated or reimplemented in
the service layer.

# Implementation Summary

- Extracted the field-validation logic that previously lived inline inside
  `repository.update_trade_signal()` into a new public function,
  `repository.validate_trade_signal_update_fields(changed_fields)`. It
  performs no database access and no serialization — only validation
  (empty/unknown/protected fields, required fields set to `None`,
  whitespace-only `symbol`/`action`, non-`Decimal` `price`). This is now the
  single source of truth for update-field validation.
- `repository.update_trade_signal()` was updated to call this new function
  instead of containing the checks inline. Its external behavior is
  unchanged, including returning `None` for a missing row (existence is not
  a field-validation concern).
- Added `TradeService.update_trade_signal(trade_signal_id, **changed_fields)`
  in `database/service.py`, which orchestrates, in this exact order:
  1. Validate `changed_fields` via
     `repository.validate_trade_signal_update_fields()`.
  2. Fetch the existing row via `repository.get_trade_signal_by_id()`,
     raising `ValueError` if no trade signal exists with that id.
  3. Write the pre-edit snapshot via `repository.create_trade_signal_edit()`,
     using `dataclasses.asdict()` of the fetched row as the full-row JSON
     snapshot.
  4. Apply the update via `repository.update_trade_signal()` and return the
     result.

  This ordering guarantees that no edit row is ever written for an update
  that will not happen (validation and existence are checked before the
  snapshot is written), and that the update never proceeds without the
  snapshot already in place.
- As with `check_duplicate_signal`, `TradeService` never commits or rolls
  back the connection; the caller owns the transaction.

# Files Changed

- `database/repository.py` (extended) — new public
  `validate_trade_signal_update_fields()` function; `update_trade_signal()`
  refactored to call it instead of duplicating the checks.
- `database/service.py` (extended) — new
  `TradeService.update_trade_signal()` method.
- `tests/test_service.py` (extended) — new
  `TradeServiceUpdateTradeSignalTests` class.

# Tests Added

`tests/test_service.py` gained `TradeServiceUpdateTradeSignalTests` (8
tests):

- Update returns the updated `TradeSignal` with the new field values.
- A single update writes exactly one `trade_signal_edits` row whose
  `previous_values` snapshot matches the pre-edit field values.
- Two sequential updates produce an ordered two-row edit history, each
  snapshot reflecting the state immediately before that specific edit.
- Updating a nonexistent `trade_signal_id` raises `ValueError` and writes
  zero edit rows.
- An empty update, an unknown field, and a non-`Decimal` price each raise
  before any edit row is written (verified via a zero-edit-row count).
- A behavioral test — `test_service_and_repository_reject_the_same_invalid_updates`
  — drives 11 invalid `changed_fields` cases (empty, unknown field,
  protected fields, required fields set to `None`, whitespace-only
  `symbol`/`action`, non-`Decimal` price) through both
  `repository.update_trade_signal()` and `TradeService.update_trade_signal()`,
  asserting both raise the same exception type and neither writes an edit
  row. This enforces the single-validation-contract requirement while
  remaining resilient to future internal refactoring (it checks shared
  behavior, not shared function identity).

**Result: 141/141 tests passing** (133 carried over from 2B.5a–2B.6a + 8 new)
via `python -m unittest discover -s tests -v`.

# Known Limitations

- `TradeService.ingest_message(...)` (2B.6c) is not yet implemented —
  `update_trade_signal()` is not yet wired into any ingestion or UI flow.
- No `edited_by` tracking, per Section 3 of the design doc (no
  authentication/user-account system exists in V1).
