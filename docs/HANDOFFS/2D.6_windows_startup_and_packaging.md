# Summary

Objective: provide a reliable Windows startup method for a non-technical
user — allowing the application to start without VS Code, a terminal, or
Claude Code open — and define dependency installation/packaging for
Version 1, without introducing new runtime dependencies or changing any
application business logic.

## Approved approach: script-based launcher, no new dependencies

A single double-clickable `start.bat` at the repo root, using a repo-local
virtual environment rather than any compiled/standalone packaging (e.g.
PyInstaller was evaluated and explicitly rejected for V1 — see the
approved planning discussion — to avoid adding new packaging dependencies
and known Streamlit-bundling friction). This keeps Version 1 consistent
with the project's "keep it simple" and "avoid unnecessary dependencies"
rules.

## `start.bat` behavior

1. Resolves its own directory via `%~dp0` and `cd /d`s into it, so it
   works correctly regardless of the working directory it's launched from
   (Explorer double-click, a desktop shortcut, a pinned taskbar icon,
   etc.).
2. **Python detection**: tries the `py -3` launcher first; falls back to
   a bare `python` only if `py -3` is unavailable. If neither resolves, a
   clear on-screen error is shown and the window pauses before exiting,
   rather than closing silently.
3. **Repo-local `.venv`**: creates `.venv` only if it does not already
   exist. Existing installations are always reused on subsequent runs —
   no activation is required or performed anywhere in the script.
4. **Dependency installation**: runs `pip install -r requirements.txt`
   only when the venv was just created, or when `streamlit` is not
   importable inside the existing venv (checked via
   `.venv\Scripts\python.exe -c "import streamlit"`). A normal second run
   against an already-provisioned venv performs no installation step at
   all.
5. **Streamlit startup**:
   ```
   .venv\Scripts\python.exe -m streamlit run app/streamlit_app.py --server.showEmailPrompt=false
   ```
   `--server.showEmailPrompt=false` suppresses Streamlit's first-run
   interactive "onboarding email" prompt (see below) without creating or
   modifying any global user configuration file
   (`%USERPROFILE%\.streamlit\credentials.toml` is never touched).
   Automatic browser opening remains Streamlit's default behavior
   (unchanged; no `--server.headless` flag is used).
6. Every failure point (Python missing, venv creation failure, dependency
   install failure, Streamlit startup failure) shows a clear, specific
   on-screen error message and pauses the window before exiting, so a
   non-technical user is never left looking at a window that flashed and
   silently closed.
7. A visible console window is an accepted, explicit V1 tradeoff (see
   Known Limitations).

## Streamlit email-prompt correction (mid-milestone review finding)

Milestone review flagged that Streamlit's one-time-per-machine
interactive "onboarding email" prompt would block startup indefinitely
for a non-technical user, since `start.bat`'s launch has no attached
interactive stdin. The smallest safe correction was applied: adding the
supported `--server.showEmailPrompt=false` command-line flag to the
`streamlit run` invocation — a one-line change, no credentials file
created or modified, browser auto-opening left untouched.

## Confirmed Streamlit import-collision root cause and structural fix

Applying the email-prompt suppression above let script execution proceed
past the point it had previously always been blocked, which surfaced a
second, previously-hidden failure:

```
ModuleNotFoundError: No module named 'app.logging_config'; 'app' is not a package
```

**Root cause, fully proven** (source-level and empirical, documented in
the investigation preceding this fix): Streamlit's own bootstrap
(`streamlit/web/bootstrap.py: _fix_sys_path()`) unconditionally inserts
`os.path.dirname(os.path.abspath(<script>))` — i.e. the directory
*containing* the target script — at `sys.path[0]`. Because the entry
point was named `app/app.py`, that inserted directory was the `app/`
folder itself, which also contains a file literally named `app.py`.
Python's import system then resolved the name `app` to that file (a
plain module, no `__path__`) instead of to the real `app` **package** at
the repo root, so `from app.logging_config import ...` failed with
exactly the observed error. This reproduced identically regardless of
venv activation state, `PYTHONPATH`, or which of the two equivalent
`streamlit run` target spellings was used — it is purely a consequence
of the script's filename colliding with its own containing directory's
name, not of `start.bat`, the package structure, or application import
logic. It never surfaced in the automated test suite because
`python -m unittest discover` never invokes Streamlit's bootstrap.

**Structural fix applied** (the lowest-risk option identified during
investigation): renamed the entry-point script so it no longer collides
with its own directory name:

```
app/app.py  ->  app/streamlit_app.py
```

`start.bat`'s `streamlit run` target and every other reference to the old
path were updated accordingly (see Files Modified). No import inside the
file, no business logic, and no other application behavior changed —
confirmed by diff: the renamed file is identical to the original except
for one self-referential docstring line.

# Known Limitations

- **Python must already be installed on the target machine.** `start.bat`
  detects Python (`py -3` first, `python` fallback) but does not install
  it; this is an accepted V1 tradeoff from the approved plan, not a bug.
- **A visible console window remains** for the duration of the app's
  runtime — an accepted V1 tradeoff; the window also carries pip-install
  output on first run and Streamlit's own server log lines throughout.
- **First startup may take noticeably longer** than later ones, since it
  creates the virtual environment and installs all dependencies from
  `requirements.txt` before Streamlit can start.
- No fixed enum/`CHECK` constraint or other unrelated schema/business
  concern was touched — carried over from prior milestones and unrelated
  to this one.
- Date/Time extraction from source messages remains unimplemented
  (carried over, unrelated to this milestone).
