2B.5c — Messages Repository (`database/repository.py`)

# Date

# Summary

Extended `database/repository.py` (built up incrementally per Milestone 2B.5)
with `raw_messages`-related data-access functions, the third slice after
2B.5a (sources) and 2B.5b (traders). As with the earlier slices, all
functions accept an existing `sqlite3.Connection` supplied by the caller;
the module never opens a connection itself and never commits or rolls back,
leaving transaction control entirely to the caller. Rows are mapped to the
existing `RawMessage` dataclass from `database/models.py` via a new private
helper, `_row_to_raw_message()`. All SQL is parameterized.

`raw_text` is stored exactly as supplied — no stripping, case changes,
Unicode normalization, or line-ending changes are performed anywhere in the
insert path, preserving the design's write-once, verbatim guarantee for
`raw_messages.raw_text`. `content_hash` is computed as
`hashlib.sha256(raw_text.encode("utf-8")).hexdigest()`, exactly as specified.
`metadata` is serialized deterministically with `json.dumps(metadata,
sort_keys=True)` on write when non-null, and deserialized with
`json.loads()` on read (returning `None` when the stored column is `NULL`);
no metadata validation schema was introduced.

This milestone is insert-and-read only, per the design's immutability
requirement for raw messages: no update, upsert, or save-or-update function
was added, and no `UPDATE raw_messages` SQL statement exists anywhere in the
module. Duplicate `external_id` handling relies entirely on the existing
database constraint — `create_raw_message` does not catch or translate
`sqlite3.IntegrityError`.

This slice covers only `raw_messages` table access, per the roadmap's
incremental plan for Milestone 2B.5 (2B.5a–2B.5e).

# Repository Functions Added

- `create_raw_message(conn, source_id, raw_text, external_id=None, metadata=None, received_at=None) -> RawMessage` —
  inserts one raw-message row and returns the complete persisted
  `RawMessage`, including the generated `id`, computed `content_hash`, and
  DB-defaulted `ingested_at` (re-read via a follow-up `SELECT` on
  `lastrowid`, since SQLite does not return DB-default column values from an
  `INSERT`); raises `sqlite3.IntegrityError` uncaught for a missing
  `source_id` or a duplicate non-null `external_id` within the same source;
  allows `TypeError` to propagate uncaught for non-JSON-serializable
  metadata.
- `get_raw_message_by_external_id(conn, source_id, external_id) -> RawMessage | None` —
  exact lookup scoped to both `source_id` and `external_id`; returns `None`
  if no row matches. Never performs a global, cross-source lookup.
- `get_raw_messages_by_content_hash(conn, content_hash) -> list[RawMessage]` —
  lookup by content hash, returning every match ordered by `id`; content
  hashes are not unique, so identical raw text remains insertable multiple
  times; returns an empty list when no rows match.

# Test Summary

`tests/test_repository_sources.py` gained a new `RawMessagesRepositoryTests`
class (20 tests) alongside the unchanged, still-passing
`SourcesRepositoryTests` (11 tests) and `TradersRepositoryTests` (16 tests)
classes from 2B.5a/2B.5b, using the standard library's `unittest` against a
temporary file-based SQLite database:

- Creating a raw message with required fields only, and with all optional
  fields.
- Generated `id` and `ingested_at`.
- Exact preservation of multiline raw text (including `\r\n` and trailing
  blank lines) and of UTF-8/non-ASCII text.
- Correct SHA-256 hash of the exact UTF-8 encoding of the raw text.
- Metadata dictionary round-trip, and `metadata=None` round-trip.
- External-ID lookup: correct match, scoped to `source_id`, and `None` when
  missing.
- Duplicate non-null `external_id` within the same source raises
  `sqlite3.IntegrityError`.
- The same `external_id` is allowed across two different sources.
- Multiple raw messages with `external_id=None` are allowed.
- Content-hash lookup: all matches returned, empty list when missing, and
  results ordered by `id` ascending.
- Identical raw text with different external IDs is insertable.
- Repository writes are not committed automatically (verified with two
  connections to the same temporary database file).
- No raw-message update function or `UPDATE raw_messages` SQL exists
  anywhere in `database/repository.py`.

**Result: 47/47 tests passing** (11 sources + 16 traders + 20 raw_messages)
via `python -m unittest discover -s tests -v`.
