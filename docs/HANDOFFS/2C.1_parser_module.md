# Milestone

2C.1 — Parser Module (`app/parser.py`)

# Objective

Implement a source-independent parser module that converts raw pasted trade
message text into structured trade-signal data, matching the fields
`TradeService.ingest_message()` expects for its `trade_signals` argument.
Per `docs/IMPLEMENTATION_ROADMAP.md`, the parser must contain no
source-specific assumptions (e.g. no Discord-specific logic) and no
networking, so that future sources (Discord, Telegram, Slack, Email, etc.)
can reuse it without modification.

# Files Implemented

- `app/parser.py` (new) — `parse_message(raw_text: str) -> list[dict]`, a
  pure function with no database, UI, `TradeService`, or networking
  dependency.
- `tests/test_parser.py` (new) — unit tests for the module.

# What Was Implemented

`parse_message()` splits raw text into lines and parses each line
independently against a documented V1 grammar:

```
ACTION SYMBOL [STRIKE][C|P|CALL|PUT] [MM/DD/YYYY] [@PRICE] [free text...]
```

- **ACTION**: `BTO`, `STC`, `BTC`, `STO`, `BUY`, `SELL` — case-insensitive,
  normalized to upper case, and preserved exactly as entered (`BUY` is never
  mapped to `BTO`; `SELL` is never mapped to `STC`).
- **SYMBOL**: 1-5 letters, normalized to upper case.
- **Option marker**: an optional strike quantity immediately followed by
  `C`/`P` (or the words `CALL`/`PUT`) sets `option_type`. The strike number
  itself is not stored — the frozen V1 schema has no strike column.
- **Expiration**: only a date with an explicit year (`MM/DD/YYYY` or
  `MM/DD/YY`, two-digit years treated as `20YY`) is recognized and
  normalized to ISO8601. A bare date with no year (e.g. `7/19`) is left
  unparsed rather than guessed, to avoid a hidden wall-clock dependency,
  consistent with `TradeService.ingest_message()`'s existing
  `reference_time` convention.
- **Price**: a token prefixed with `@` or `$`, parsed as a `Decimal` — never
  a float.
- **Position size**: any remaining tokens on the line, joined verbatim.

A line that does not start with a recognized action keyword, or has no
symbol token, contributes no signal and is silently skipped — `parse_message`
never raises on malformed input.

# Known Limitations

- The V1 grammar was designed without example raw trade-alert messages (none
  were available in project docs); it is a reasonable, documented starting
  grammar, not validated against real-world message samples. It may need
  revision once real messages are available.
- Strike price is recognized in the option marker token but not stored
  anywhere — the frozen V1 schema has no strike column.
- A date with no year is intentionally left unparsed (`expiration` stays
  `None`); the raw date text falls into `position_size` instead of being
  lost.
- `parse_message()` is not yet wired into any UI or `TradeService` flow —
  that is explicitly Milestone 2C.2/2C.3's job, not this one.
