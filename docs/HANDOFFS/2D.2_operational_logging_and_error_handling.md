# Summary

Objective: add persistent application logging and a safe UI error
boundary to the Streamlit manual-entry application, using only the Python
standard library, so operational failures produce useful diagnostic logs
while non-technical users see concise, safe messages — without exposing
stack traces, exception message text, raw trade content, trader
identifiers, or filesystem paths in the UI or in the logs themselves.

`app/logging_config.py` (new) provides two independently idempotent
handler-configuration functions attached to a logger tree rooted at
`discord_traders`:

- `configure_console_logging()` — attaches exactly one console
  (`StreamHandler`) handler, identified by a private marker attribute
  (not a generic "any handler exists" check). Called once at `app/app.py`
  module top level, so console logging is available from application
  startup. Sets the default level to `INFO`.
- `configure_file_logging()` — attaches exactly one
  `logging.handlers.RotatingFileHandler`, identified by its own,
  independent private marker attribute, so an existing console handler
  never blocks file-handler attachment and an existing file handler is
  never duplicated. Always calls `configure_console_logging()` first, so
  console fallback is guaranteed even if this function is invoked
  independently. Configured lazily — called only at the start of
  `app/app.py`'s Submit-to-Database workflow, not at module import time —
  specifically so that Streamlit's `AppTest` harness, which re-executes
  `app.py`'s entire module body on every rerun, never triggers real file
  I/O during Parse-only test interactions or in any test that doesn't
  itself click Submit.

Both functions are exercised on every Streamlit rerun (the harness
re-runs `app.py`'s module body on each interaction); the private-marker
design guarantees repeated calls, across any number of reruns, never
attach a second console or file handler.

`resolve_log_path()` resolves the runtime log-file location, fully
independent of `database.config.resolve_database_path()` and
`DISCORD_TRADERS_DB_PATH` — neither is read or called anywhere in
`app/logging_config.py`. Precedence:

1. A non-empty `DISCORD_TRADERS_LOG_PATH`, after stripping surrounding
   whitespace, normalized via `Path(value).expanduser().resolve()` — both
   absolute and relative override values resolve to a normalized
   absolute path.
2. `%LOCALAPPDATA%\DiscordTraders\logs\discord_traders.log`, if
   `LOCALAPPDATA` is set to a non-empty value after stripping whitespace.
3. `Path.home()\AppData\Local\DiscordTraders\logs\discord_traders.log`,
   used only when `LOCALAPPDATA` is missing or empty/whitespace-only.

`RotatingFileHandler` is configured with exactly: `maxBytes=1_000_000`,
`backupCount=3`, `encoding="utf-8"`, `mode="a"`, `delay=True`.

Log format: `"%(asctime)s %(levelname)-8s %(name)s: %(message)s"`, with
`datefmt="%Y-%m-%d %H:%M:%S"`.

`app/app.py`'s Submit-workflow exception boundary is broadened from
`(ValueError, TypeError, sqlite3.Error)` to also include `OSError`, plus a
final `except Exception` safety net — both branches log at `ERROR` via
`log_operation_failure()` and display the same unchanged, fixed message:
`"Could not save the message to the database."`. A new guard around
`parse_message()` distinguishes an unexpected parser exception
(`"Could not parse this message."`, `ERROR`-logged) from the unchanged,
valid empty-result message (`"No trade signals found in this message."`).

# Notes

## Confirmations

- Python standard-library `logging` only — no new dependency.
- Logger tree rooted at `discord_traders`; console and file handlers are
  independently idempotent, each identified by its own private marker
  attribute, so neither's presence affects the other's attachment.
- Console logging is configured at application startup (module top
  level); file logging is configured lazily, only at the start of the
  Submit workflow.
- Default level `INFO`; format `"%(asctime)s %(levelname)-8s %(name)s: %(message)s"`;
  `datefmt="%Y-%m-%d %H:%M:%S"`.
- `RotatingFileHandler(maxBytes=1_000_000, backupCount=3, encoding="utf-8", mode="a", delay=True)`.
- `resolve_log_path()` is fully independent of
  `database.config.resolve_database_path()`/`DISCORD_TRADERS_DB_PATH` —
  neither is read anywhere in `app/logging_config.py`.
- File-logging initialization failures (any `Exception`, not only
  `OSError`, though never `BaseException`) never block submission;
  console logging remains available as fallback in every case.
- A partially constructed file handler (e.g. one whose formatter
  assignment fails after construction succeeds) is explicitly closed
  before being discarded — no partial or unmarked handler is ever left
  attached.
- Streamlit reruns — which re-execute `app.py`'s entire module body on
  every interaction — never duplicate either handler.
- `sanitized_traceback()` retains, per stack frame, only the source
  file's basename (never the absolute path or any parent directory
  component), line number, function name, and the static source-code
  line, followed only by the exception's class name.
- Logs exclude, under all current and hypothetical future exception
  content: exception message text (`str(exc)`/`repr(exc)`), absolute
  source paths, raw trade message text, trader display names, external
  trader IDs, parsed signal field values, the resolved database path,
  the resolved log path, any environment-variable value, and SQL values
  or parameters.
- Unexpected parser failure: `"Could not parse this message."` Valid
  empty parse result (unchanged): `"No trade signals found in this
  message."` Submission failure (unchanged): `"Could not save the
  message to the database."`
- Both known operational failures (`ValueError`, `TypeError`,
  `sqlite3.Error`, `OSError`) and unexpected failures (the final
  `except Exception` safety net) are logged at `ERROR`.
- No `CRITICAL` events are introduced anywhere in this milestone.
- Duplicate-signal advisories remain a normal, non-exceptional business
  result, surfaced via `st.warning(...)` exactly as before, and are
  logged only at `WARNING` (count only) — never at `ERROR`.
- Rollback and unconditional connection cleanup (`finally: if conn is not
  None: conn.close()`) remain intact and unchanged, now also covering the
  broadened exception classes.
- Parsed review state (`st.session_state["parsed_signals"]`/
  `["parsed_raw_text"]`) remains available after a persistence failure —
  no new state-clearing logic was introduced.
- `TradeService.ingest_message()` remains the sole UI persistence entry
  point; `app/app.py` still never calls `database/repository.py`
  directly.
- No schema, model, parser, repository, service, database-config, or
  database-initialization redesign occurred.
- No backup, recovery, stored-signal review UI, correction workflow,
  packaging, Discord integration, telemetry, or third-party dependency
  work was included — all remain explicitly out of scope for 2D.2.
- Architecture (`docs/DATABASE_DESIGN_V1.md`) and the database design
  remain FROZEN; unchanged throughout this milestone.
