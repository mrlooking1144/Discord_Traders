# Milestone

2B.6a — TradeService Scaffold + Duplicate Detection (`database/service.py`)

# Objective

Create `database/service.py` exposing a `TradeService` class, and implement
the soft, advisory duplicate check described in Section 5 of
`docs/DATABASE_DESIGN_V1.md`: matching `trader_id + symbol + action +
option_type + price + expiration` within a short ingestion window, as a
method on the service. Per the roadmap's acceptance criterion, the check
never raises or blocks on the duplicate determination itself — it returns a
warning the caller can act on. This is the first slice of Milestone 2B.6
(`service.py`, implemented incrementally); edit-on-update wiring (2B.6b) and
the `ingest_message` entry point (2B.6c) are explicitly out of scope.

`TradeService` is the first module permitted to contain business rules,
per the roadmap's framing of Milestone 2B.6 as "the only layer that
orchestrates multiple repository calls and contains business rules." All
SQL remains inside `database/repository.py`, per Product Owner decision
during design review — `service.py` never runs SQL directly.

# TradeService Behavior

`TradeService` is constructed with a caller-owned `sqlite3.Connection`
(`TradeService(conn)`), stored on `self.conn`. It never commits or rolls
back the connection, matching the caller-owns-the-transaction convention
already established throughout `repository.py`. Its one public method,
`check_duplicate_signal(trader_id, symbol, action, option_type, price,
expiration, reference_time)`, orchestrates a single call to the new
repository function and translates the result into the advisory return
value described below. It contains no SQL of its own.

# Duplicate-Matching Rules

Per `DATABASE_DESIGN_V1.md` Section 5, an existing trade signal is
considered a possible duplicate when all of the following match the
candidate exactly:

- `trader_id`
- `symbol`
- `action`
- `option_type` (including `None` matching `None`, for non-option trades)
- `price` (compared as its stored decimal string; `Decimal("3.25")` and
  `Decimal("3.250")` are **not** considered equal, since price is never
  rounded or normalized beyond `Decimal`'s own string representation,
  consistent with 2B.5d)
- `expiration` (including `None` matching `None`)

No fuzzy or partial matching is performed — Section 8 of the design doc
explicitly defers fuzzy/ML-based duplicate matching out of scope for
Version 1.

# 5-Minute Backward-Looking Window

`DUPLICATE_WINDOW_MINUTES = 5` is defined as a module-level constant in
`database/service.py`. Given a caller-supplied `reference_time`, the window
compared against `trade_signals.created_at` is:

```
window_start = reference_time - 5 minutes
window_end   = reference_time
```

This is a backward-looking window (per Product Owner confirmation): it asks
"was this same trade already recorded in the last 5 minutes?", not a
symmetric window around `reference_time`.

`reference_time` is a required `str` argument in the same format SQLite's
`CURRENT_TIMESTAMP` produces (`"YYYY-MM-DD HH:MM:SS"`), passed explicitly by
the caller rather than read from wall-clock `now()`, so both tests and
later ingestion (2B.6c) are deterministic. It is parsed with an explicit
`datetime.strptime` format string rather than `datetime.fromisoformat`, to
avoid Python-version-dependent parsing of the space-separated (non-`T`)
timestamp format.

# Inclusive Boundaries

Both `window_start` and `window_end` are inclusive: a signal with
`created_at` exactly equal to either boundary counts as a match. This is
implemented via SQL's `BETWEEN`, which is inclusive on both ends.

# Validation Behavior

Per Product Owner decision, "basic invalid inputs may raise `ValueError` or
`TypeError`; the duplicate result itself must never raise or block":

- `trader_id` must not be `None`; `symbol` and `action` must not be `None`,
  empty, or whitespace-only — all raise `ValueError`. (Validated inside
  `get_trade_signals_matching`, not duplicated in `TradeService`.)
- `price` must be a `Decimal` or `None` — any other type (`float`, `str`,
  `int`) raises `TypeError`.
- `reference_time` must be a non-empty string in the expected timestamp
  format — missing or malformed values raise `ValueError`, validated
  directly in `TradeService.check_duplicate_signal` before any query runs.
- Once all inputs pass validation, the presence or absence of a duplicate
  is reported purely via the `str | None` return value — zero matching
  rows, or no existing signals at all, returns `None` without raising
  anything.

# Repository Function Added

`get_trade_signals_matching(conn, trader_id, symbol, action, option_type,
price, expiration, window_start, window_end) -> list[TradeSignal]` —
purely mechanical field- and time-range matching, with no knowledge of
"duplicate," "advisory," or "window" semantics; those belong entirely to
`TradeService`. Reuses the existing `_serialize_price` helper for
consistent `Decimal`-only price handling, matching `create_trade_signal`'s
existing validation. Returns rows ordered by `id` ascending, matching the
convention of every other list-returning repository function.
