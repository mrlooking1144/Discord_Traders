# Milestone

2B.6c — `TradeService.ingest_message(...)` (`database/service.py`)

# Objective

Implement the single public ingestion entry point on `TradeService`: given a
source, raw message text/metadata, and zero or more parsed trade signals,
persist everything in the correct order (`sources` → `traders` →
`raw_messages` → `trade_signals`) via `database/repository.py`, surfacing
duplicate warnings from 2B.6a. Per the roadmap's acceptance criterion,
`ingest_message()` is the only entry point the UI layer will call to persist
data — no other code reaches into `repository.py` directly for a full
ingestion.

This is the final slice of Milestone 2B.6 (`service.py`, implemented
incrementally); the scaffold and advisory duplicate check (2B.6a) and the
edit-on-update rule (2B.6b) were already in place.

# What Was Implemented

`TradeService.ingest_message()` orchestrates, in this exact linear order:

1. Resolve the source via `repository.get_or_create_source()`.
2. Resolve the trader (get-or-create when `external_trader_id` is supplied;
   always create when it is not — see Trader Identity Behavior below).
3. Insert the raw message via `repository.create_raw_message()`.
4. For each parsed trade signal, in list order: call
   `self.check_duplicate_signal()` (2B.6a) against the resolved trader, then
   insert it via `repository.create_trade_signal()` regardless of the
   duplicate result — matching 2B.6a's "never blocks" rule.
5. Return a plain `dict` bundling everything created.

No new `repository.py` functions were needed — every step reuses an
existing repository function. `database/models.py` and
`database/schema.sql` were not touched.

# Design Decisions

Per Product Owner review of the design proposal, the following were
mandated up front (superseding the design doc's open questions):

- **No new return type.** The design proposal's `IngestResult` dataclass was
  rejected. `ingest_message()` returns a plain `dict` with keys `source`,
  `trader`, `raw_message`, `trade_signals`, `duplicate_warnings`.
- **Trader identity never matches by display name.** If
  `external_trader_id` is given, an existing trader is reused when one
  already exists for `(source_id, external_trader_id)`; otherwise a new
  trader is created with that external ID. If `external_trader_id` is not
  given, a new trader row is **always** created — `get_traders_by_name()` is
  never consulted by this method, since duplicate display names are
  explicitly allowed by the schema and are not a reliable identity signal
  (Design Doc §3).
- **No raw-message duplicate pre-check.** `ingest_message()` does not call
  `get_raw_message_by_external_id()` before inserting. The hard,
  DB-enforced `UNIQUE` constraint on `(source_id, external_id)` is left to
  do its job; `sqlite3.IntegrityError` propagates to the caller unchanged
  on a collision.
- **`reference_time` has no hidden clock.** It is a required argument,
  passed straight through to `check_duplicate_signal()` per call. There is
  no `datetime.now()` fallback anywhere in this method, consistent with
  2B.6a's determinism requirement.
- **Editing an already-ingested Discord message is out of scope.**
  `ingest_message()` only handles new-message ingestion. It never calls
  `TradeService.update_trade_signal()` (2B.6b). `raw_messages.raw_text`
  remains write-once, per Design Doc §6.
- **Strictly linear orchestration**, with no branching beyond the
  trader-identity get-or-create/always-create split and the per-signal
  loop: Source → Trader → Raw Message → (for each parsed signal:
  `check_duplicate_signal()` → `create_trade_signal()`) → Return.

As with `check_duplicate_signal()` and `update_trade_signal()`,
`TradeService` never commits or rolls back the connection; the caller owns
the transaction.

# Public API

```python
def ingest_message(
    self,
    source_name: str,
    trader_name: str,
    raw_text: str,
    reference_time: str,
    external_trader_id: str | None = None,
    external_message_id: str | None = None,
    metadata: dict | None = None,
    received_at: str | None = None,
    trade_signals: list[dict] | None = None,
) -> dict:
```

**Args:**

- `source_name` — Source type name (e.g. `'discord'`); looked up or created.
- `trader_name` — Display name/handle as seen in the source.
- `raw_text` — The original message, verbatim.
- `reference_time` — The timestamp to check each parsed signal's duplicate
  window against, in the same format as `trade_signals.created_at`
  (`"YYYY-MM-DD HH:MM:SS"`). Always supplied by the caller; never read from
  wall-clock `now()`.
- `external_trader_id` — Stable source-provided trader ID, when available.
- `external_message_id` — Source-provided message ID, when available.
- `metadata` — Opaque JSON-serializable metadata, or `None`.
- `received_at` — ISO8601 timestamp of when the source sent the message.
- `trade_signals` — Zero or more dicts of parsed trade signal fields
  (`symbol`, `action`, `option_type`, `price`, `expiration`,
  `position_size`). `raw_message_id` and `trader_id` are filled in by this
  method, not the caller.

**Returns:** a `dict` with exactly these keys:

- `"source"` — the resolved `Source`.
- `"trader"` — the resolved `Trader`.
- `"raw_message"` — the newly created `RawMessage`.
- `"trade_signals"` — `list[TradeSignal]`, in the same order as the
  `trade_signals` argument.
- `"duplicate_warnings"` — `list[str | None]`, one entry per created trade
  signal, from `check_duplicate_signal()`.

**Raises:**

- `ValueError` — if `source_name` or `trader_name` is empty or
  whitespace-only; if `reference_time` is missing or malformed; or if any
  parsed signal is missing a required field (`symbol`, `action`).
- `TypeError` — if `price` on any parsed signal is supplied and is not a
  `Decimal`.
- `sqlite3.IntegrityError` — if `external_message_id` collides with an
  already-ingested message for this source, or `external_trader_id`
  collides in a race.

# Orchestration Flow

```
source = get_or_create_source(source_name)
    ↓
trader = get-or-create (external_trader_id given)
       | always create (external_trader_id absent)
    ↓
raw_message = create_raw_message(source.id, raw_text, ...)
    ↓
for each parsed signal, in order:
    warning = check_duplicate_signal(trader.id, symbol, action,
                                      option_type, price, expiration,
                                      reference_time)
    signal  = create_trade_signal(raw_message.id, trader.id, symbol,
                                   action, option_type, price,
                                   expiration, position_size)
    ↓
return {source, trader, raw_message, trade_signals, duplicate_warnings}
```

# Duplicate Handling Behavior

- **Raw-message level (hard, DB-enforced):** no pre-check. A second insert
  with the same `(source_id, external_id)` raises `sqlite3.IntegrityError`
  from the schema's partial `UNIQUE` index; `ingest_message()` does not
  catch it.
- **Trade-signal level (soft, advisory, from 2B.6a):** every parsed signal
  is checked via `check_duplicate_signal()` before it is inserted, and is
  inserted regardless of the result — nothing is ever blocked. Because the
  check runs against the same connection, a signal within the *same*
  `ingest_message()` call can be flagged as a duplicate of an
  already-inserted (uncommitted) signal earlier in the same `trade_signals`
  list; this is intentional (a message that lists the same trade twice
  legitimately warns on the second occurrence) and is covered by a test.
- **Edited messages are out of scope.** A re-ingested Discord message with
  the same `external_id` but different text has no defined handling in this
  method (see Design Decisions above) — it hits the raw-message
  `IntegrityError` path like any other duplicate `external_id`.

# Trader Identity Behavior

- `external_trader_id` given → `get_trader_by_external_id()`; reuse if
  found, otherwise `create_trader()` with that external ID.
- `external_trader_id` not given → `create_trader()` unconditionally. Trader
  display names are never used to look up or match an existing trader, per
  Design Doc §3 (names can collide or change; only a stable source-provided
  ID is a reliable identity signal). A direct consequence: two messages from
  the same person without a captured external ID will always create two
  separate `traders` rows.

# Validation Behavior

No new validation logic lives in `ingest_message()` itself — it delegates
entirely to the repository functions it calls, consistent with 2B.6a/2B.6b:

- `source_name` / `trader_name` empty or whitespace-only → `ValueError`
  from `get_or_create_source()` / `create_trader()`.
- `reference_time` missing or not matching `"YYYY-MM-DD HH:MM:SS"` →
  `ValueError` from `check_duplicate_signal()`.
- A parsed signal missing `symbol` or `action` (or blank) → `ValueError`
  from `create_trade_signal()`. Signal dict fields are read with `.get()`,
  so a missing key behaves identically to an explicit `None` value.
- A parsed signal's `price` not a `Decimal` → `TypeError` from
  `create_trade_signal()` (and from `check_duplicate_signal()`, which runs
  first).
- `metadata` not JSON-serializable → `TypeError` from
  `create_raw_message()`.

# Tests Added

`tests/test_service.py` gained `TradeServiceIngestMessageTests` (17 tests):

- New message with zero signals persists the message only, with empty
  result lists.
- New message with one signal and no prior duplicate returns
  `duplicate_warnings == [None]`.
- A signal matching a prior signal (different `ingest_message()` call, same
  `external_trader_id`) within the window still persists, with a non-`None`
  warning.
- Multiple signals in one call with mixed duplicate/non-duplicate results
  get correct per-index warnings.
- Two identical signals within the *same* message: the first gets `None`,
  the second is flagged against the first.
- `external_trader_id` reuses the existing trader across two calls; no
  `external_trader_id` always creates a new trader across two calls.
- Created signals' `raw_message_id` and `trader_id` match the returned
  `raw_message`/`trader`.
- `metadata` round-trips through `raw_messages.metadata`.
- Empty/whitespace `source_name` and `trader_name` each raise `ValueError`
  and write nothing.
- Missing and malformed `reference_time` each raise `ValueError`.
- Non-`Decimal` `price` on a signal raises `TypeError`.
- A signal missing `symbol` raises `ValueError`.
- Duplicate `external_message_id` for the same source raises
  `sqlite3.IntegrityError`.
- `ingest_message()` does not commit pending writes on the connection.

One test-construction note: `trade_signals.created_at` and
`raw_messages.ingested_at` are populated by SQLite's `CURRENT_TIMESTAMP`,
which is **UTC**. Tests that exercise duplicate detection through
`ingest_message()` (as opposed to `check_duplicate_signal()` directly, which
takes an explicit `created_at` fixture) compute `reference_time` from
`datetime.now(timezone.utc)` with a small forward buffer, not local wall
time, to reliably bracket the real inserted rows.

# Known Limitations

- No transactional/rollback guarantees are added by this method itself —
  a failure partway through (e.g. the second of three signals raising)
  leaves the source, trader, raw message, and any already-inserted signals
  uncommitted but still present in the connection's pending transaction,
  exactly as `repository.py` calls would behave individually. Verifying
  atomic all-or-nothing behavior against a real `db.py` connection is
  explicitly Milestone 2B.7's job, not this one.
- Editing an already-ingested Discord message (same `external_id`, changed
  text) has no defined path anywhere in the system yet — flagged during
  design review as a gap, not addressed here.
- `ingest_message()` is not yet wired into any UI or parser flow (Streamlit
  wiring is explicitly out of scope for Milestone 2B per the roadmap).
