initialization, connections, service orchestration, and SQLite persistence
are all genuine, not mocked. The single narrow exception is the rollback
test, which patches only `database.service.create_trade_signal` to
manufacture a controlled mid-transaction failure after earlier real writes
have already occurred, in order to exercise `app.py`'s real rollback path.

# Tests Executed

8 new real-SQLite integration tests, each against its own unique temporary
SQLite database (`tempfile.mkstemp`) with real schema initialization
performed before any UI interaction:

1. `test_full_manual_entry_persists_all_tables_with_correct_relationships`
   — verifies persisted rows and foreign-key relationships across
   `sources`, `traders`, `raw_messages`, and `trade_signals`: source name
   is `"manual"`; trader display name and stable external trader ID are
   stripped correctly; raw message text is stored verbatim; parsed symbol,
   action, option type, price, expiration, and position size are
   persisted correctly; `raw_messages.source_id`, `trade_signals.raw_message_id`,
   and `trade_signals.trader_id` all reference the correct persisted rows;
   exactly one row exists per table.
2. `test_same_external_trader_id_reuses_single_trader_row_across_two_messages`
   — two different valid messages submitted with the same stable external
   trader ID resolve to a single `traders` row, with both trade signals
   referencing that same trader.
3. `test_trader_display_name_not_updated_on_resubmission_with_different_name`
   — documents current `TradeService.ingest_message()` behavior: reusing
   an existing trader by external ID never renames it, so a later
   submission with a different display name does not change the stored
   name.
4. `test_duplicate_signal_within_window_persists_both_with_advisory_warning`
   — a first matching signal is seeded deterministically via a direct
   `TradeService` call (fixed field values, committed before the UI
   interaction), then a second matching signal is submitted through the
   real UI path. Verifies: one `manual` source, one trader, two raw
   messages (both referencing the same source), two trade signals (both
   referencing the same trader, both carrying the expected duplicate-key
   fields), the second signal actually persisted despite the warning, and
   exactly one advisory `"Possible duplicate"` warning displayed — never
   blocking, never reimplemented in the UI.
5. `test_malformed_input_persists_no_rows_and_blocks_submission` —
   unparseable pasted text produces no reviewed signal, no trader fields
   or Submit button ever render, and all four tables remain at zero rows.
6. `test_successful_submission_commits_and_is_durable_via_fresh_connection`
   — a successful submission's data is visible through a brand-new SQLite
   connection opened after the app's own submission connection has closed,
   proving durability through a real commit.
7. `test_mid_transaction_failure_rolls_back_leaving_no_partial_rows` — with
   `database.service.create_trade_signal` patched to raise after the
   source, trader, and raw message have already been written for real
   (uncommitted) on the real connection, `app.py`'s real rollback path
   leaves zero rows in all four tables, verified via a fresh connection.
8. `test_production_database_file_never_created_or_modified` — asserts
   the repository-root `discord_traders.db` is byte-for-byte unchanged
   (or still absent) after a full successful submission.

Additionally, every test in the file (via the shared
`_ManualEntryIntegrationTestCase` fixture) independently snapshots
`discord_traders.db` (existence, size, mtime, SHA-256) before and after,
so protection against creating or modifying the production database is
enforced across all 8 tests, not just test 8.

# Known Limitations

- The duplicate-warning test's second submission relies on `app.py`'s
  internally computed `reference_time` (real wall-clock `datetime.now()`
  at Submit time) falling within the 5-minute duplicate window relative to
  the seeded first signal's real, database-assigned `created_at`. Neither
  value is overridable from application code today (`trade_signals.created_at`
  is `DEFAULT CURRENT_TIMESTAMP` with no repository-level override, and
  `app.py`'s second-submission `reference_time` is a non-patchable literal
  wall-clock call, same class of constraint as `_DB_PATH`). The test
  structurally guarantees the seeded signal's timestamp precedes the
  second submission's (seed is committed before `AppTest` starts), leaving
  only a ~300-second safety margin against ordinary in-process test
  execution time — not a production-code limitation, just a documented,
  low-risk residual wall-clock dependency.
- Date/Time extraction from source messages remains unimplemented (carried
  over, unrelated to this milestone).
