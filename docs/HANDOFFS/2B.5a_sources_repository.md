# Milestone

2B.5a — Sources Repository (`database/repository.py`)

# Summary

Implemented source-related data-access functions in a new `database/repository.py`
module, the first slice of the incrementally-built Milestone 2B.5 repository
layer. The module accepts an existing `sqlite3.Connection` supplied by the
caller; it never opens a connection itself and never commits or rolls back,
leaving transaction control entirely to the caller. Rows are mapped to the
existing `Source` dataclass from `database/models.py` — no dictionaries or raw
`sqlite3.Row` objects are returned. All SQL is parameterized.

Source name validation (rejecting empty and whitespace-only names) is
centralized in a single private helper, `_validate_source_name()`, used by
both public functions so validation behavior is consistent across the module.
Validation raises the built-in `ValueError` — no new exception class or
hierarchy was introduced, since `database/models.py` (Milestone 2B.4) defines
no exception types to reuse. This deviation from the original task wording was
raised to the Product Owner and resolved before implementation began.

This slice covers only `sources` table access, per the roadmap's incremental
plan for Milestone 2B.5 (2B.5a–2B.5e).

# Repository Functions Added

- `get_source_by_name(connection, name) -> Optional[Source]` — exact-name
  lookup via parameterized SQL; returns `None` if no match; raises
  `ValueError` for an empty or whitespace-only name.
- `get_or_create_source(connection, name) -> Source` — returns the existing
  `Source` if one with the exact name exists, otherwise inserts and returns
  the newly created `Source`; raises `ValueError` for an empty or
  whitespace-only name; relies on the database's `UNIQUE` constraint on
  `sources.name` as the authoritative uniqueness guarantee.

# Tests Implemented

`tests/test_repository_sources.py` (new; `tests/` previously empty), using
the standard library's `unittest` against a temporary file-based SQLite
database (no new test-framework dependency added):

- Creating a new source.
- Retrieving an existing source by exact name.
- Returning `None` for a missing source.
- Calling `get_or_create_source` twice with the same name returns the same
  ID and leaves exactly one row.
- Rejecting an empty name in `get_or_create_source`.
- Rejecting a whitespace-only name in `get_or_create_source`.
- Rejecting an empty name in `get_source_by_name`.
- Rejecting a whitespace-only name in `get_source_by_name`.
- Confirming no row is created after invalid input.
- Confirming a valid name with surrounding whitespace is preserved exactly
  (not trimmed or normalized).
- Confirming repository functions do not autocommit: using two connections
  to the same temporary database file, the second connection cannot see an
  inserted row until the first connection commits, and can see it
  afterward.

All 11 tests pass via `python -m unittest discover -s tests -v`.

# Known Limitations

- No exception class exists in `database/models.py` for validation errors;
  `get_source_by_name` and `get_or_create_source` raise the built-in
  `ValueError` instead. If a project-wide validation exception is introduced
  later, this module will need to be revisited.
- Only `sources` table access exists in `repository.py` so far — traders,
  raw_messages, trade_signals, and trade_signal_edits functions are not yet
  implemented (2B.5b–2B.5e).
- No service layer (`TradeService`), parser, or Discord/Streamlit integration
  exists yet; `repository.py` has no callers besides its own tests.
