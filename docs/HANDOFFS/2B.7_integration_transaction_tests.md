# Milestone

2B.7 — Integration and transaction tests (`tests/`)

# Objective

End-to-end tests exercising `TradeService` against a real, temp-file SQLite
database (`database/db.py` + `database/schema.sql`, exactly as production
code will use them), confirming transactional integrity: `TradeService`
never commits or rolls back on its own, so a caller's `commit()` /
`rollback()` decisions are what actually determine what is persisted. This
is a test-only milestone, per `docs/IMPLEMENTATION_ROADMAP.md` §3 — no new
production module was expected or added.

# What Was Implemented

A single new test file, `tests/test_integration_transactions.py`, adding
one test case class, `IntegrationTransactionTests`, with 5 tests. Each test
uses its own temp-file SQLite database (`tempfile.mkstemp` +
`initialize_database()` + `get_connection()`), matching the fixture pattern
already used in `tests/test_service.py`. No other file was created or
modified.

1. **`test_full_ingest_happy_path_commits_and_persists`** — a full
   `ingest_message()` call with two trade signals, `commit()`, then
   verification via a second, concurrently open connection: row counts
   across all four tables (`sources`, `traders`, `raw_messages`,
   `trade_signals`) and correct FK linkage
   (`trade_signals.raw_message_id`/`trader_id`, `raw_messages.source_id`).
2. **`test_partial_failure_rollback_leaves_no_partial_rows`** — a
   two-signal `ingest_message()` call where the second signal is missing
   `symbol` and raises `ValueError` mid-loop, after the source, trader, raw
   message, and first signal were already inserted (uncommitted) on the
   same connection. After the caller catches the exception and calls
   `rollback()`, all four table counts are asserted at zero, checked both
   on the same connection and on a second, independent connection.
3. **`test_duplicate_warning_non_blocking_persists_both_signals`** — two
   identical signals in one `ingest_message()` call: the first gets a
   `None` duplicate warning, the second gets a non-`None` warning, and
   after `commit()` both rows are confirmed persisted via a second
   connection — proving the advisory duplicate check never blocks a write.
4. **`test_edit_history_after_multiple_corrections`** — an ingested signal
   is corrected twice via `TradeService.update_trade_signal()`
   (`price`, then `option_type`), followed by a single `commit()`. A
   second connection reads `trade_signal_edits` and confirms two rows in
   order, with each JSON snapshot matching the pre-edit state at that
   point (first snapshot's `price` is `"3.25"`; second snapshot's `price`
   is `"3.50"` and `option_type` is `"call"`), and the current row
   reflects both corrections (`price` `"3.50"`, `option_type` `"put"`).
5. **`test_commit_durability_via_fresh_connection`** — after
   `ingest_message()` and `commit()`, the original connection is closed
   entirely (not just left open alongside a second one). A brand-new
   connection is then opened against the same database file and confirms
   all rows are still present and correct — proving durability to disk,
   not just visibility to a second simultaneously-open connection.

# Design Decisions

Per the approved design review, the implementation scope was narrowed from
the original 7-scenario proposal to exactly the 5 tests above. Two
scenarios proposed during design review were explicitly cut as outside the
frozen Milestone 2B.7 roadmap and are **not** implemented:

- Rollback mid-edit (a failure inside `update_trade_signal()` itself,
  before the edit snapshot is written).
- Duplicate `external_message_id` rollback (the `sqlite3.IntegrityError`
  path from the `raw_messages` `UNIQUE` constraint).

No production code changes were made or needed. No defect was found in
`TradeService`, `repository.py`, or `db.py` during testing; all observed
behavior (caller-owned commit/rollback, advisory non-blocking duplicate
warnings, edit-on-update ordering) matched what was already documented in
the 2B.6a–2B.6c handoffs.
