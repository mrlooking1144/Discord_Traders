# Objective

Wire the manual entry UI's parsed structured trade data (Milestone 2C.1
`app.parser.parse_message()`, reviewed per Milestone 2C.2) to
`TradeService.ingest_message()`, so a user-submitted, parsed message is
persisted through the existing service layer only, surfacing any duplicate
warnings returned to the user. Per `docs/IMPLEMENTATION_ROADMAP.md`,
`TradeService.ingest_message()` remains the only persistence entry point;
`app/app.py` makes no direct calls to `database/repository.py`.

# Implemented UI Flow

Two-step workflow, preserved from Milestone 2C.2 and not collapsed into one
action:

1. **Parse Message** — invokes `parse_message()` on the pasted raw text and
   displays the structured signal(s) for review (unchanged from 2C.2).
2. **Review** — the parsed signals remain visible for inspection before any
   persistence is attempted.
3. **Submit to Database** — a separate button that, only once a valid parse
   exists, persists the reviewed signals through `TradeService`.

# Required Inputs

Two new required UI fields, rendered once a valid parse exists:

- **Trader display name** — free text, required; submission is blocked with
  a controlled error if left blank.
- **Stable external trader ID** — free text, required; submission is
  blocked with a controlled error if left blank. Required (not optional) so
  repeat manual entries from the same real trader reuse the same `traders`
  row instead of fragmenting across a new row on every submission, which
  would otherwise silently defeat `TradeService`'s trader-scoped duplicate
  detection.

# Persistence Path

A fresh, non-cached database lifecycle is created on every Submit click (no
`st.cache_resource`, no module-level or session-cached connection):

```
config = DatabaseConfig(db_path="discord_traders.db")
initialize_database(config)
conn = get_connection(config)
service = TradeService(conn)
result = service.ingest_message(...)
```

# Transaction Behavior

- `conn.commit()` is called inside the same protected block as
  `ingest_message()`, so a commit failure is caught the same way as a
  validation or persistence failure — success is never reported before a
  commit has actually succeeded.
- On any caught exception (including a commit failure), `conn.rollback()`
  is attempted only if a connection was actually created; the rollback
  attempt itself is wrapped so a rollback failure cannot raise past the
  handler.
- The connection is always closed in a `finally` block, on both the
  success and failure paths.
- If connection creation itself fails (`initialize_database()` or
  `get_connection()` raises), no connection exists, so no rollback is
  attempted and nothing is closed beyond what was actually opened.

# Fixed Metadata Decisions

Approved for manual entry in this milestone:

- `source_name`: fixed to `"manual"`.
- `external_message_id`: always `None` (no stable message ID exists for
  pasted text).
- `metadata`: always `None`.
- `received_at`: always `None` (source message date/time extraction is
  deferred per `docs/DATABASE_DESIGN_V1.md` Section 8).
- `reference_time`: captured once per Submit click, in UTC, formatted as
  `%Y-%m-%d %H:%M:%S` (the exact format `TradeService.ingest_message()`
  requires).
- `trader_name` and `external_trader_id` are `.strip()`-ed before being
  passed to the service; `raw_text` is passed exactly as reviewed, never
  stripped or otherwise transformed.

# Duplicate Behavior

- `TradeService.ingest_message()`'s `duplicate_warnings` are displayed via
  `st.warning` for every non-`None` entry.
- Duplicate warnings remain advisory only: they never prevent persistence,
  and the UI implements no duplicate-detection logic of its own — it only
  surfaces what the service layer already returned.

# Stale-Preview Protection

- On every rerun, the current textarea content is compared against the raw
  text that produced the stored preview (`st.session_state`); if they
  differ, the stored parsed signals and raw text are cleared immediately,
  removing the review/Submit UI until the text is parsed again.
- A failed or empty reparse (`parse_message()` returns no signals) clears
  any previously stored valid preview, so a user cannot paste unparseable
  text, reparse, and still submit an older valid result.
- `st.session_state` is used for exactly two values: the parsed signals and
  the exact raw text used to produce them — nothing else.

# Testing Added

All persistence-path tests patch `database.db.initialize_database`,
`database.db.get_connection`, and `database.service.TradeService` as
controlled test doubles; no real SQLite database is touched (reserved for
Milestone 2C.4). Coverage includes:

- Successful submission calling `ingest_message()` exactly once, with
  correctly mapped parser signals and required metadata.
- Successful-creation display and duplicate-warning display (non-blocking).
- Unparseable input never reaching the service.
- Missing required trader fields blocking submission before the service is
  called.
- Service exceptions (`ValueError`), database initialization failures, and
  connection failures each displaying a controlled error with no raw
  exception text or traceback exposed.
- Commit success and commit failure, including that a commit failure
  attempts rollback, does not display a success message, and still closes
  the connection.
- Rollback occurring after a post-connection failure, and explicitly not
  occurring when connection creation itself fails.
- Connection closure after both success and failure.
- Stale-preview clearing on edited raw text and on a failed reparse.
- Trader name and external trader ID passed to the service with leading/
  trailing whitespace stripped, while the raw message stays verbatim.
