# Summary

Objective: add controlled, timestamped backups of the production SQLite
database, validate backup integrity, and define safe restore behavior that
prevents partial or destructive restore operations — without changing the
frozen architecture, database design, or `TradeService`'s role as the sole
persistence entry point.

## Backup creation

`database/backup.py` (new) provides `create_backup(db_path, backup_dir=None,
*, target_subdir=None, apply_retention=True)`:

- Uses `sqlite3.Connection.backup()`, not a raw filesystem copy. This
  produces a consistent, standalone backup file regardless of whether the
  source database's journal mode is the default rollback journal or WAL,
  and is safe to call even while a connection is currently open on the
  source (exercised directly by
  `tests/test_backup.py::CreateBackupTests`, including a dedicated WAL-mode
  source test).
- Writes to a temporary file (`<final-name>.tmp`) in the destination
  directory, verifies the result with `PRAGMA integrity_check` via
  `_verify_integrity()`, and only then publishes it under its final name
  via an atomic `os.replace()`. A failed integrity check removes the
  temporary file and leaves no final file behind.
- Backup directory resolution (`resolve_backup_dir()`) mirrors
  `database.config.resolve_database_path()`'s precedence: a non-empty
  `DISCORD_TRADERS_BACKUP_PATH` override (normalized via
  `Path(value).expanduser().resolve()`), else
  `%LOCALAPPDATA%\DiscordTraders\backups`, else
  `Path.home()\AppData\Local\DiscordTraders\backups`. The directory is
  created lazily, only at backup time.
- Filenames are UTC-timestamped: `discord_traders_YYYYMMDD_HHMMSS_ffffff.db`
  (matching the UTC convention `app/app.py` already uses for
  `reference_time`). An unexpected collision with an already-existing final
  filename is refused (`FileExistsError`), never silently overwritten.
- Retention keeps the latest 10 verified backups (`_MAX_RETAINED_BACKUPS`),
  deleting the oldest first, and runs only after a new backup has been
  successfully created and verified — never after a failed or unverified
  attempt. A single backup's deletion failure during retention is
  non-fatal (logged as a sanitized `WARNING`); the newly created backup is
  still retained and the operation is still reported as successful.
  Temporary (`.tmp`) and unrecognized/invalid filenames are excluded from
  retention's count entirely.
- `target_subdir` and `apply_retention=False` support the pre-restore
  safety backup (see Restore, below): publishing under a `prerestore_safety`
  subdirectory keeps it out of normal retention's top-level directory scan
  until it is explicitly promoted.
- `list_backups(backup_dir=None)` returns published, correctly named
  backups (oldest first), excluding temporary/invalid files.

`app/app.py` adds one new, minimal, unconditional "Create Backup" button
(always rendered, not tied to the Parse/Submit review state) that calls
`database.backup.create_backup(resolve_database_path())`, following the
same try/except/log/fixed-message pattern as the existing Submit workflow:
`configure_file_logging()` at the start, a fixed generic
`"Database backup created successfully."` on success, a fixed generic
`"Could not create a database backup."` on failure, and
`log_operation_failure()` for sanitized `ERROR` logging. No restore control
of any kind is present in the UI.

## Restore (CLI-only)

`restore_backup(db_path, candidate_path, backup_dir=None)` is invoked only
via `python -m database.backup restore <backup_file_path>` (the module's
`main()`), and is never called from `app/app.py`. `main()` prints an
explicit precondition notice before proceeding: the Streamlit application
and every other program with the production database open must be fully
closed first. This is a documented precondition, reinforced (not proven
with absolute certainty across every external process) by a best-effort
exclusivity probe.

Order of operations, in full:

1. **Candidate validation**, before any modification: SQLite file-header
   magic-byte check, a read-only open, `PRAGMA integrity_check`, and a
   required-tables-and-columns schema-compatibility check
   (`_verify_schema_compatibility()`) against the frozen schema's exact
   table and column set (`sources`, `traders`, `raw_messages`,
   `trade_signals`, `trade_signal_edits`) — a presence check only, never a
   migration.
2. **Best-effort exclusive-access probe** (`_probe_exclusive_access()`),
   when the production database exists: a `BEGIN EXCLUSIVE`/`COMMIT` with a
   short busy timeout, raising `ExclusiveAccessError` if another connection
   appears to hold the database.
3. **Pre-restore safety backup**, when the production database exists:
   created via the same `create_backup()` used for normal backups,
   published under `prerestore_safety/` and excluded from normal retention
   until the restore, including post-restore validation, fully succeeds.
4. **Sidecar quarantine**: any existing `-wal`, `-shm`, or `-journal` file
   beside the production database is moved into a quarantine directory
   before the main file is touched. **If any sidecar cannot be safely
   read, moved, or quarantined, the restore hard-aborts here** —
   `SidecarQuarantineError` is raised, the production database and all of
   its sidecars are left completely untouched, and the now-unused
   pre-restore safety backup is removed. Any sidecars already quarantined
   earlier in the same attempt are moved back before the error is raised,
   so quarantine failure is all-or-nothing.
5. **Atomic main-file replacement**: the validated candidate is copied to a
   temporary file beside the production path, then published via
   `os.replace()`.
6. **Post-restore validation**: `PRAGMA integrity_check` and the same
   required-table/required-column schema check are re-run against the
   now-replaced production file.
7. **Recovery on post-restore validation failure**: the protected
   pre-restore safety backup (not the newly, unsuccessfully restored file)
   is restored into place via the same atomic-replace mechanism, its
   quarantined sidecars are moved back beside it, and the original
   exception is re-raised. The failed candidate's data is never relied
   upon once it fails this validation.
8. **Stale-sidecar cleanup**: only once post-restore validation has
   succeeded are the quarantined sidecars discarded — never before, so
   step 7 can still recover them if validation fails. No stale sidecar is
   ever left beside a successfully restored database.
9. **Safety-backup promotion**: on full success, the pre-restore safety
   backup is moved into the main backups directory (making it a normal,
   retention-eligible backup) and retention is applied.

# Known Limitations

- The exclusive-access probe (`_probe_exclusive_access()`) is best-effort:
  it reliably detects a genuinely locked/actively-written database via a
  `BEGIN EXCLUSIVE` attempt with a short busy timeout, but cannot guarantee
  detection of every possible external-process scenario on Windows with
  absolute certainty. This is an approved design tradeoff, not an
  oversight — restore's primary safety requirement is the documented
  precondition that the application and all other database users be fully
  stopped first.
- A database connection opened for any reason (including the exclusive-
  access probe or the safety-backup step) can cause SQLite itself to clean
  up a genuinely stale, non-live `-wal`/`-journal` sidecar as part of its
  own hot-journal recovery behavior. This does not affect restore's
  correctness (quarantine always completes before any further connection
  is opened against the production path after replacement), but it was an
  important, empirically-confirmed consideration in designing the sidecar
  quarantine/recovery tests.
- Recovery after a post-restore validation failure restores the original
  database's logical content exactly (confirmed via table-level content
  checks) but is not guaranteed to be byte-for-byte identical to the
  original file, since both the safety backup and its restoration go
  through `sqlite3.Connection.backup()`'s own page-level copy mechanism
  rather than a raw file copy.
- No automated test spawns the actual `python -m database.backup restore
  ...` command as a subprocess; `restore_backup()` and `main()`'s argument
  handling are exercised directly as Python functions, consistent with how
  the rest of this codebase tests CLI-adjacent logic.
- Date/Time extraction from source messages remains unimplemented
  (carried over, unrelated to this milestone).

- `sqlite3.Connection.backup()` is used for all backup creation — never a
  raw filesystem copy — producing a consistent, standalone backup file
  regardless of the source's rollback-journal or WAL journal mode.
- Default backup directory: `%LOCALAPPDATA%\DiscordTraders\backups\`,
  overridable via `DISCORD_TRADERS_BACKUP_PATH`, with a `Path.home()`
  fallback identical in structure to `resolve_database_path()`'s and
  `resolve_log_path()`'s existing precedence. The backup directory is
  created lazily, only at backup time.
- Filename format: `discord_traders_YYYYMMDD_HHMMSS_ffffff.db`, UTC-based,
  with unexpected collisions refused rather than overwritten.
  Publication always goes through a temporary file, a
  `PRAGMA integrity_check`, and an atomic rename.
- Retention keeps the latest 10 verified backups, oldest deleted first,
  only after a successful verified backup; deletion failures are
  non-fatal and sanitized; temporary/invalid files never count toward
  retention.
- `app/app.py` gained exactly one minimal, always-rendered "Create Backup"
  control; no restore control of any kind exists in the UI.
- Restore is CLI-only (`python -m database.backup restore <path>`) and is
  never reachable from the Streamlit application. The application and all
  other database users must be fully stopped before running it — this is
  a documented precondition, reinforced by a best-effort exclusive-access
  probe, not a UI-enforced guarantee.
- Restore candidates are validated (SQLite header, read-only open,
  integrity check, required-table-and-column schema compatibility) before
  any modification to the production database is made.
- A pre-restore safety backup is always created and verified before
  replacement (when a current database exists), and is protected from
  normal retention until post-restore validation succeeds.
- Existing `-wal`, `-shm`, and `-journal` sidecars are quarantined before
  the main database file is replaced; if any cannot be safely quarantined,
  the restore hard-aborts, leaving the production database and all
  sidecars completely untouched.
- The main database file is replaced atomically; post-restore integrity
  and schema validation runs immediately afterward, and a failure there

  successfully restored database.
