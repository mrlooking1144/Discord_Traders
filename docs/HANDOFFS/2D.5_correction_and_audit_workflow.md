# Summary

Objective: allow controlled corrections to persisted trade signals, routed
through `TradeService.update_trade_signal()`, preserving the existing
`trade_signal_edits` audit mechanism, and displaying that audit history
read-only — without changing the frozen architecture, database design, or
any existing repository function.

## Backward-compatible `update_trade_signal()` extension

`TradeService.update_trade_signal()` gained one new optional, keyword-only
parameter:

```python
def update_trade_signal(
    self,
    trade_signal_id: int,
    expected_current_values: dict | None = None,
    **changed_fields,
) -> TradeSignal:
```

- **Legacy mode** (`expected_current_values is None`): byte-for-byte the
  pre-existing Milestone 2B.6b behavior — sparse partial updates, no
  six-field shape requirement, no no-op check, no stale-conflict check.
  Every existing 2B.6b test in `tests/test_service.py` continues to pass
  unmodified, confirming this is a true additive extension, not a
  redesign.
- **Controlled-correction mode** (`expected_current_values` supplied):
  `changed_fields` and `expected_current_values` must each contain
  **exactly** the six approved fields — `symbol`, `action`, `option_type`,
  `price`, `expiration`, `position_size` — rejecting `raw_message_id`,
  `trader_id`, `id`, `created_at`, `updated_at`, and any missing or extra
  field. This closes a gap the planning review specifically flagged: UI
  hiding of those fields alone would not have stopped a caller from
  passing them; the service layer now enforces the correction-specific
  allowed-field set directly.

### Check order (controlled-correction mode)

1. Validate `changed_fields`' key shape (exactly the six approved fields).
2. Validate `expected_current_values`' key shape (the same six fields).
3. Run the existing, unchanged `repository.validate_trade_signal_update_fields()`
   for type/content correctness (`Decimal` price, non-empty symbol/action).
4. Fetch the current signal via `get_trade_signal_by_id()`.
5. Raise `TradeSignalNotFoundError` (new, a `ValueError` subclass, used
   only in controlled-correction mode) if it no longer exists — the
   legacy path's own missing-signal `ValueError` is completely unchanged.
6. Build the canonical typed current-value snapshot from the fetched row.
7. Compare `expected_current_values` to that snapshot — raise
   `StaleTradeSignalError` (new) on any mismatch. **This runs before the
   no-op check.**
8. Compare `changed_fields` to that snapshot — raise a plain `ValueError`
   if identical (a no-op).
9. Only after both checks pass: write exactly one full-row
   `trade_signal_edits` snapshot (unchanged mechanism), then apply the
   update.

No stale, missing, invalid, or no-op correction ever creates an audit row
or changes `updated_at`.

### Typed comparison — never float, never string-vs-Decimal

The canonical six-field representation used throughout (comparisons and
the current-value snapshot) is: `symbol: str`, `action: str`,
`option_type: str | None`, `price: Decimal | None`, `expiration: str |
None`, `position_size: str | None`. `price` is parsed from the persisted
decimal string into `Decimal` before every comparison, so no comparison
ever holds a raw string against a `Decimal`, and no float is ever
involved anywhere in this path.

## Audit history: `TradeService.list_trade_signal_audit_history()`

New, additive, read-only method. Calls the existing, unchanged
`get_trade_signal_edits()`, decodes each row's `previous_values` JSON,
validates the decoded value is a `dict` (raising the new
`AuditHistoryError` for malformed or non-dict JSON), and returns a
**newest-first** list of plain dicts containing exactly `id`, `edited_at`,
and the six previous editable values — `price` preserved as its exact
stored string, never converted to float. The raw `previous_values` JSON
text itself is never returned or logged; `app/app.py` never decodes audit
JSON itself.

## UI: "Correct Signal" inside the existing Review Signals workflow

Entirely inside the existing `Review Signals` sidebar branch — no new
navigation entry, no multipage redesign:

- **"Correct Signal"** button (visible once a signal is selected) enters
  correction mode, capturing the signal ID and the canonical typed
  six-field snapshot into `st.session_state`.
- **Correction form**, prefilled from that snapshot:
  - Symbol — text input, normalized to **uppercase**.
  - Action — `st.selectbox` restricted to **BTO, STC, BTC, STO, BUY,
    SELL**, normalized to uppercase.
  - Option type — `st.selectbox` restricted to **blank/`None`, call,
    put**, normalized to lowercase (or `None` when blank).
  - Price — text input, parsed to `Decimal` (blank → `None`).
  - Expiration — text input, validated as `YYYY-MM-DD` (blank → `None`).
  - Position size — text input (blank → `None`).
- **Required confirmation checkbox** gates Save; clicking Save without it
  checked shows the validation message and never calls `TradeService` at all.
- **Client-side syntactic validation** (parsing price/expiration,
  requiring the checkbox) happens before the correction service call —
  **not** before any connection is opened, since the `Review Signals`
  branch already has one open at that point for the review list and audit
  history. Malformed input never reaches `TradeService.update_trade_signal()`
  and creates no audit row.
- **Save/Cancel** and the **read-only "Correction History"** section all
  share the **same connection already opened** for the review list within
  that rerun — no extra connection is opened, and the database is never
  initialized here. The Milestone 2D.4 no-implicit-creation guarantee is
  fully preserved: if the database does not yet exist, the branch still
  never calls `initialize_database()`, never opens a connection, and shows
  the same fixed `"No trade signals found."` message.
- **Session-state rules**: entering correction mode stores
  `correction_signal_id` and `correction_expected_values`; changing the
  selected signal, changing filters so the signal disappears from the
  current result set, a stale/not-found conflict, or clicking Cancel all
  clear both keys; a validation failure keeps the form open; a successful
  save clears both keys before the next rerun (the corrected values become
  visible on that next rerun, exactly like the existing Submit workflow's
  committed changes already do — not retroactively within the same script
  pass).
- **Raw message remains read-only** (`disabled=True`) throughout; **no
  delete control** exists anywhere in this workflow.

### Exact fixed messages

| Condition | Message |
|---|---|
| Validation failure or no-op | `"Please enter a valid correction that changes at least one field."` |
| Conflict — stale or not found | `"This trade signal changed or is no longer available. Reload it before correcting."` |
| Persistence failure | `"Could not save the trade signal correction."` |
| Success | `"Trade signal correction saved."` |
| Audit-history load failure | `"Could not load correction history."` |

None of these ever include field values, previous values, filesystem
paths, or exception text.

## Logging

Reuses the existing sanitized `discord_traders` logger tree and
`log_operation_failure()` contract exactly, unchanged since Milestone
2D.2: fixed operation labels plus the exception's class name and a
sanitized traceback only. Never logged: raw message/field values (current
or previous), trader/source names, the raw `previous_values` JSON,
exception message text, or filesystem paths. No `CRITICAL` logging is
introduced anywhere in this milestone.

# Known Limitations

- No fixed enum/`CHECK` constraint exists at the database layer for
  `action`/`option_type` (the schema remains frozen and unchanged); the
  approved vocabulary is enforced only at the application/service layer
  via the correction UI's `st.selectbox` controls, consistent with the
  frozen design's existing "free text, application-validated" approach
  for these columns.
- The "two browser sessions" stale-conflict scenario is tested at the
  `TradeService` layer directly against real SQLite (two sequential calls
  sharing one connection, the second using an intentionally stale
  snapshot) rather than simulating two literally concurrent Streamlit
  sessions/connections — the clearest and most deterministic way to
  express the scenario.
- Date/Time extraction from source messages remains unimplemented
  (carried over, unrelated to this milestone).
