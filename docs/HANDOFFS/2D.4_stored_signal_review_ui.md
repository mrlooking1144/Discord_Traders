# Summary

Objective: display persisted trade signals, support practical filtering
by source, trader, symbol, and date, and show the original raw message
alongside the persisted parsed fields — strictly read-only, with all
reads routed through the existing `TradeService` boundary — without
changing the frozen architecture, database design, or any existing
repository/service function.

## Sidebar workflow navigation

`app/app.py` gained a single navigation control at the top of the script:

```python
workflow = st.sidebar.radio("Workflow", ["Manual Message Entry", "Review Signals"])
```

The entire script body is restructured into a plain Python `if`/`elif` on
this value — **not** `st.tabs()`. Streamlit executes the code inside
every `st.tabs()` block on every rerun regardless of which tab is
visually selected, so tabs would not have prevented the Review workflow's
query (or the Manual Message Entry workflow's parser/submission/backup
calls) from running on every single rerun. With a plain `if`/`elif`,
only the branch matching the selected value executes at all — confirmed
by dedicated tests (`WorkflowNavigationTests`) proving each workflow's
functions are never invoked while the other is selected.

The existing Parse → review → Submit workflow and the Milestone 2D.3
"Create Backup" control moved, unchanged in behavior, into the
`Manual Message Entry` branch — same widgets, same messages, same
logging, same `TradeService.ingest_message()` call site. Create Backup is
not shown and cannot be triggered from `Review Signals`.

## Review Signals workflow (strictly read-only)

**No implicit database creation.** Before touching the database in any
way, the branch resolves the path and checks existence directly:

```python
db_path = resolve_database_path()
if not Path(db_path).exists():
    st.info("No trade signals found.")
else:
    ...
```

When the file does not exist, `initialize_database()` is never called,
no connection of any kind is opened (which would otherwise implicitly
create the file, since `sqlite3.connect()` creates a new empty database
on open), and no parent directory is created — the screen simply shows
`"No trade signals found."` and stops. The existing Manual Message Entry
Submit workflow remains the only code path responsible for database
initialization before its first write; this is unchanged from Milestones
2D.1–2D.3.

When the file already exists, the branch opens a connection via
`database.db.get_connection()` (never `initialize_database()`),
constructs a `TradeService`, calls
`TradeService.list_trade_signals_for_review()` with the current filter
values, and always closes the connection in `finally`. Zero matching
rows show the same fixed `"No trade signals found."` message. Any
unexpected `sqlite3.Error`/`OSError` during connection or query is
caught, logged via the existing sanitized `log_operation_failure()`, and
shown as the fixed message `"Could not load stored trade signals."`.

## Repository: `get_trade_signals_for_review()`

New, additive function in `database/repository.py`. Joins:

```sql
FROM trade_signals
JOIN traders       ON trade_signals.trader_id      = traders.id
JOIN raw_messages  ON trade_signals.raw_message_id = raw_messages.id
JOIN sources       ON raw_messages.source_id       = sources.id
```

returning, per row: `id`, `symbol`, `action`, `option_type`, `price`,
`expiration`, `position_size`, `created_at`, `updated_at`, `source_name`,
`trader_name`, `external_trader_id`, `raw_text`. `price` is returned as
the exact stored decimal string, never converted to float.

- **Ordering**: `ORDER BY trade_signals.id DESC` — deterministic
  newest-first, using the existing primary-key/rowid index; no new index
  required or added (schema frozen).
- **Default limit**: `LIMIT 100`, applied in SQL, before any Python-side
  processing.
- **Filters, every one applied in SQL before `LIMIT`**:
  - `source_name` — exact match (`sources.name = ?`).
  - `trader_name` — exact match (`traders.name = ?`).
  - `symbol` — case-insensitive exact match
    (`UPPER(trade_signals.symbol) = UPPER(?)`), never a substring match.
  - `date` — filters only `trade_signals.created_at` (never
    `raw_messages.received_at`), with a private `_date_range_bounds()`
    helper computing an **inclusive** start (`D 00:00:00`) and an
    **exclusive** next-day boundary (`D+1 00:00:00`), using the same
    `"%Y-%m-%d %H:%M:%S"` timestamp format already stored by
    `create_trade_signal()`/`CURRENT_TIMESTAMP`.
  - Blank or `None` filters omit their `WHERE` fragment entirely — never
    a wildcard/always-true placeholder.
- All SQL is parameterized; no string-formatted user values reach the
  query. No existing repository function was changed.

## Service: `TradeService.list_trade_signals_for_review()`

New, additive method in `database/service.py`. A thin, read-only
delegation: uppercases a non-blank `symbol` filter, then calls
`get_trade_signals_for_review()` and returns its result unchanged. No
business rules, no writes, no orchestration beyond that one
normalization step. `TradeService` remains the application's sole
persistence/service boundary; `app/app.py` still never imports or calls
`database.repository` directly, and still executes no raw SQL anywhere.

## Display

- **Summary table**: `st.dataframe`, showing signal ID, source, trader,
  symbol, action, option type, price (exact stored string), expiration,
  and created timestamp — newest first. The raw message text is
  deliberately **not** included in this table.
- **Detail selection**: a plain `st.selectbox` of signal IDs (not
  `st.dataframe`'s row-selection API), driving a read-only detail panel
  showing every persisted structured field, source/trader context, the
  external trader ID only when non-`NULL`, and the raw message text —
  the raw message appears **only** in this detail view.
- **Nullable fields** (`option_type`, `price`, `expiration`,
  `position_size`, `external_trader_id`) are displayed safely (blank or
  omitted), never causing an error.
- **Duplicate-warning status is omitted** — it is never persisted
  anywhere in the schema (`check_duplicate_signal()` produces an
  advisory string only at ingestion time, never written to any table),
  so it cannot be shown as stored data.
- **No edit, correction, delete, or audit-history controls** of any kind
  exist in this workflow — confirmed by tests asserting zero buttons
  render and no mutating function is ever reachable.

## Logging

Query/connection failures log through the existing `discord_traders`
logger tree via `log_operation_failure()`, exactly as the Submit and
Create Backup workflows already do: a fixed operation label and the
exception's class name only, via `sanitized_traceback()` — never filter
values, symbols, trader names, source names, raw message text, exception
message text, or filesystem paths. No `CRITICAL` logging is introduced.
