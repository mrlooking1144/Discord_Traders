# Milestone

2B.5e — Trade Signal Edits Repository (`database/repository.py`)

# Objective

Extend `database/repository.py` with an insert function writing a full-row
JSON snapshot into `trade_signal_edits`, and a lookup function returning a
signal's edit history — the fifth and final slice of Milestone 2B.5, after
2B.5a (sources), 2B.5b (traders), 2B.5c (raw_messages), and 2B.5d
(trade_signals). Per the roadmap's acceptance criterion for this milestone,
this module only records edits; it does not decide *when* an edit occurs.
`update_trade_signal` (2B.5d) is unchanged and still does not write to
`trade_signal_edits` — wiring "always snapshot before update" is deferred to
`TradeService` in Milestone 2B.6b.

As with the earlier slices, both functions accept an existing
`sqlite3.Connection` supplied by the caller; the module never opens a
connection itself and never commits or rolls back, leaving transaction
control entirely to the caller. Rows are mapped to the existing
`TradeSignalEdit` dataclass from `database/models.py` via a new private
helper, `_row_to_trade_signal_edit()`. All SQL is parameterized.

# Repository Functions Added

- `create_trade_signal_edit(conn, trade_signal_id, previous_values) -> TradeSignalEdit` —
  inserts one edit-snapshot row and returns the complete persisted
  `TradeSignalEdit`, including the generated `id` and DB-defaulted
  `edited_at` (re-read via a follow-up `SELECT` on `lastrowid`).
- `get_trade_signal_edits(conn, trade_signal_id) -> list[TradeSignalEdit]` —
  returns every edit row for the given `trade_signal_id`, ordered by `id`
  ascending (chronologically); returns an empty list when none exist.
- `_row_to_trade_signal_edit(row) -> TradeSignalEdit` — private row-mapping
  helper, following the same pattern as `_row_to_trade_signal`,
  `_row_to_raw_message`, and `_row_to_trader`.

# Validation Behavior

`create_trade_signal_edit` enforces, in application code, ahead of any
database constraint:

- `trade_signal_id` must not be `None` — raises `ValueError`.
- `previous_values` must be a `dict` — any other type, including `None`,
  raises `TypeError`.
- `previous_values` must not be an empty `dict` — raises `ValueError`. This
  rule was added after Product Owner review of the initial implementation,
  which had accepted `{}`.
- No schema is enforced on the dictionary's contents — the design doc's
  "no per-field diff logic" intent means the repository does not know or
  care what keys/values a snapshot contains.
- A `trade_signal_id` that does not reference an existing `trade_signals`
  row raises `sqlite3.IntegrityError`, left uncaught and untranslated, per
  the pattern established in every earlier 2B.5 slice.

# JSON Snapshot Behavior

`previous_values` is accepted only as a Python `dict` — never a
pre-serialized JSON string — and is serialized with
`json.dumps(previous_values, sort_keys=True)` before insert, matching the
deterministic serialization already used for `raw_messages.metadata` in
2B.5c. `TradeSignalEdit.previous_values` (the field on the returned/stored
model) remains the serialized JSON string, unchanged from its
`database/models.py` definition. Nested dictionaries and lists within
`previous_values` round-trip correctly through `json.dumps`/`json.loads`.
Two edits with identical dictionary content serialize to identical strings
and are both insertable — there is no uniqueness constraint on snapshot
content.

# Append-Only Behavior

No update or delete function was added for `trade_signal_edits`, and no
`UPDATE trade_signal_edits` or `DELETE FROM trade_signal_edits` SQL exists
anywhere in `database/repository.py`, consistent with its role as an
immutable audit trail. This is verified by an automated test that both
scans the module's public function names and inspects its source text.

# Testing Summary

`tests/test_repository_sources.py` gained a new
`TradeSignalEditsRepositoryTests` class (18 tests) alongside the unchanged,
still-passing `SourcesRepositoryTests`, `TradersRepositoryTests`,
`RawMessagesRepositoryTests`, and `TradeSignalsRepositoryTests` classes from
2B.5a–2B.5d, using the standard library's `unittest` against a temporary
file-based SQLite database:

- Creating an edit from a dict, and confirming generated `id`/`edited_at`.
- Deterministic JSON serialization with sorted keys (two dicts with the same
  keys in different insertion order serialize identically).
- Nested dictionaries and lists round-trip correctly through storage and
  retrieval.
- An empty dict raises `ValueError` (corrected per Product Owner review; see
  below).
- `None` and all non-dict values (`str`, `list`, `int`) raise `TypeError`.
- `trade_signal_id=None` raises `ValueError`.
- An invalid foreign key raises `sqlite3.IntegrityError`.
- Multiple edits for the same signal are returned ordered by `id` ascending.
- Results are scoped to the requested `trade_signal_id` (edits for a
  different signal are excluded).
- A signal with no edits returns an empty list.
- Identical snapshots may be inserted more than once, each with a distinct
  `id`.
- Repository writes are not committed automatically (verified with two
  connections to the same temporary database file).
- No update or delete function or SQL exists for `trade_signal_edits`.
- `update_trade_signal` (2B.5d) still does not create any history row.

One existing 2B.5d test, `test_no_trade_signal_edits_row_created`, was
narrowed: its assertion that no `INSERT INTO trade_signal_edits` text
existed *anywhere in the module* was scoped down to scanning only
`update_trade_signal`'s own source (via
`inspect.getsource(repository.update_trade_signal)`), since that string now
legitimately exists elsewhere in the module, inside `create_trade_signal_edit`.
The test's original intent — that `update_trade_signal` itself never writes
history — is preserved and still passes.

During Product Owner review, `create_trade_signal_edit` was corrected to
reject an empty dict with `ValueError` (it had initially accepted `{}`). The
test `test_empty_dict_is_accepted` was replaced with
`test_empty_dict_raises_value_error`.

**Result: 100/100 tests passing** (11 sources + 16 traders + 20 raw_messages
+ 35 trade_signals + 18 trade_signal_edits) via
`python -m unittest discover -s tests -v`.
