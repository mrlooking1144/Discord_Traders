# Summary

Objective: replace the working-directory-relative database path in
`app/app.py` with an explicitly resolved Windows application-data path, so
production startup no longer depends on the repository or shell working
directory, while preserving `DatabaseConfig.db_path` as a required field
with no hidden default and keeping development, test, and production
database locations isolated from one another.

`database/config.py` gains a new, independently testable
`resolve_database_path()` function implementing this exact precedence:

1. A non-empty `DISCORD_TRADERS_DB_PATH` value, after stripping
   surrounding whitespace. Normalized via
   `Path(value).expanduser().resolve()` — both absolute and relative
   override values return a normalized, absolute path; a relative
   override is interpreted relative to the current working directory at
   the moment `resolve_database_path()` runs, and the returned value is
   never left cwd-relative.
2. `Path(LOCALAPPDATA) / "DiscordTraders" / "discord_traders.db"`, if
   `LOCALAPPDATA` is set to a non-empty value after stripping surrounding
   whitespace — i.e. exactly
   `%LOCALAPPDATA%\DiscordTraders\discord_traders.db` on Windows. This
   branch is **not** additionally passed through `.resolve()`.
3. `Path.home() / "AppData" / "Local" / "DiscordTraders" /
   "discord_traders.db"`, used only when `LOCALAPPDATA` is missing or
   empty/whitespace-only. This branch is also **not** additionally passed
   through `.resolve()`.

`app/app.py` removes its hardcoded `_DB_PATH = "discord_traders.db"`
constant and instead calls
`DatabaseConfig(db_path=resolve_database_path())` at the same call site —
`app.py` still explicitly resolves the runtime path and passes it into
`DatabaseConfig`; no default was added to the dataclass.

`database/db.py`'s `initialize_database()` gains one line —
`Path(config.db_path).parent.mkdir(parents=True, exist_ok=True)` —
immediately before it opens its connection, so required parent
directories are created safely. No try/except was added around it:
directory-creation failures propagate normally, since friendly runtime
error handling and operational logging are Milestone 2D.2 scope, not
2D.1. `get_connection()` and `resolve_database_path()` remain free of any
directory-creation responsibility, and `resolve_database_path()` performs
no filesystem writes at all (a deliberate property — `app.py`'s module
body, including this call, re-executes on every Streamlit
`AppTest.run()`, so the path-resolution function itself must stay
side-effect-free to keep the test suite from ever touching a developer's
real `%LOCALAPPDATA%`).

# Tests Executed

**`tests/test_config.py`** (12 new tests):
- Absolute override is used as given (already normalized).
- Relative override resolves to an absolute path anchored at the cwd.
- `~`-prefixed override is expanded via `expanduser()`.
- Redundant path segments (`a/../b/...`) are normalized away by `.resolve()`.
- Surrounding whitespace on an override is stripped before resolution.
- Empty-string override falls back to the default.
- Whitespace-only override falls back to the default.
- Default resolves to exactly `<LOCALAPPDATA>\DiscordTraders\discord_traders.db`.
- Missing `LOCALAPPDATA` uses the `Path.home()` fallback.
- Empty `LOCALAPPDATA` uses the `Path.home()` fallback.
- Whitespace-only `LOCALAPPDATA` uses the `Path.home()` fallback.
- Return value is a `str` suitable for `DatabaseConfig(db_path=...)`.

All via `unittest.mock.patch.dict(os.environ, ...)` and a patched
`database.config.Path.home` — no real `LOCALAPPDATA` or home directory is
ever read or written.

**`tests/test_db.py`** (4 new tests):
- `initialize_database()` creates a missing single-level parent directory.
- `initialize_database()` creates missing nested parent directories.
- `initialize_database()` does not fail when the parent already exists.
- Schema initialization still succeeds (all 5 designed tables present)
  after parent-directory creation.

All via `tempfile.TemporaryDirectory()` only.

**`tests/test_app.py`** (1 new test):
- `test_database_config_receives_resolved_path_from_resolve_database_path`
  — patches `database.config.resolve_database_path` to return a sentinel
  path and `database.config.DatabaseConfig` to a `MagicMock`, drives the
  app through Parse → review → Submit, and asserts `DatabaseConfig` was
  called with `db_path` equal to exactly that sentinel — direct proof of
  the wiring, not inferred from the separate real-SQLite integration
  suite.
