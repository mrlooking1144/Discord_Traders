# Milestone

2B.5d — Trades Repository (`database/repository.py`)

# Objective

Extend `database/repository.py` with insert, lookup, and partial-update
functions for `trade_signals`, the fourth slice of Milestone 2B.5 after 2B.5a
(sources), 2B.5b (traders), and 2B.5c (raw_messages). As with the earlier
slices, all functions accept an existing `sqlite3.Connection` supplied by the
caller; the module never opens a connection itself and never commits or
rolls back, leaving transaction control entirely to the caller. Rows are
mapped to the existing `TradeSignal` dataclass from `database/models.py` via
a new private helper, `_row_to_trade_signal()`. All SQL is parameterized.

Required fields (`raw_message_id`, `trader_id`, `symbol`, `action`) are
enforced in application code ahead of the schema's own constraints, and
`price` is accepted only as a `Decimal` (or `None`) and stored as its exact
decimal string, per Section 3 of `docs/DATABASE_DESIGN_V1.md` — binary
floating-point is never used for trade prices. `update_trade_signal` applies
a partial update via `**changed_fields`, rejects unknown or protected field
names (`id`, `created_at`, `updated_at`), re-validates required fields when
they are included in the update, and always bumps `updated_at` to
`CURRENT_TIMESTAMP` on a successful update. Per the roadmap's incremental
plan, this milestone does not touch `trade_signal_edits` — no audit-snapshot
row is written on update; that is the scope of Milestone 2B.5e.

# Repository Methods Added

- `create_trade_signal(conn, raw_message_id, trader_id, symbol, action, option_type=None, price=None, expiration=None, position_size=None) -> TradeSignal` —
  inserts one trade-signal row and returns the complete persisted
  `TradeSignal`, including the generated `id`, DB-defaulted `created_at`, and
  `updated_at` (re-read via a follow-up `SELECT` on `lastrowid`); raises
  `ValueError` for a missing `raw_message_id`/`trader_id` or an empty/
  whitespace-only `symbol`/`action`; raises `TypeError` if `price` is
  supplied and is not a `Decimal`; raises `sqlite3.IntegrityError` uncaught
  for a `raw_message_id` or `trader_id` that does not reference an existing
  row.
- `get_trade_signal_by_id(conn, trade_signal_id) -> TradeSignal | None` —
  exact lookup by primary key; returns `None` if no row matches.
- `update_trade_signal(conn, trade_signal_id, **changed_fields) -> TradeSignal | None` —
  applies a partial update to one or more of `raw_message_id`, `trader_id`,
  `symbol`, `action`, `option_type`, `price`, `expiration`, `position_size`
  (optional fields may be explicitly set to `None`); always bumps
  `updated_at`; returns `None` if no row exists with this id; raises
  `ValueError` if `changed_fields` is empty, names an unknown or protected
  field, or sets a required field to an invalid value; raises `TypeError` if
  `price` is supplied and is not a `Decimal`. Never writes to
  `trade_signal_edits`.

# Testing Summary

`tests/test_repository_sources.py` gained a new `TradeSignalsRepositoryTests`
class (35 tests) alongside the unchanged, still-passing
`SourcesRepositoryTests`, `TradersRepositoryTests`, and
`RawMessagesRepositoryTests` classes from 2B.5a/2B.5b/2B.5c, using the
standard library's `unittest` against a temporary file-based SQLite
database. New coverage includes:

- Creating a trade signal with required fields only, and with all optional
  fields.
- Generated `id`, `created_at`, and `updated_at`.
- Required- and optional-field round-trips via `get_trade_signal_by_id`.
- `Decimal` price stored and returned as its exact decimal string; `float`,
  `str`, and `int` prices are all rejected with `TypeError`.
- Missing `raw_message_id`/`trader_id`, and empty/whitespace-only
  `symbol`/`action`, all rejected with `ValueError`.
- A `raw_message_id` or `trader_id` that does not reference an existing row
  raises `sqlite3.IntegrityError`.
- Lookup by existing id returns the matching row; lookup by missing id
  returns `None`.
- Partial update changes only the supplied fields and leaves others intact.
- Update bumps `updated_at` (verified deterministically by pre-setting
  `updated_at` to a fixed past value rather than sleeping across a
  `CURRENT_TIMESTAMP` tick).
- Optional fields can be explicitly updated to `None`.
- Update rejects `None` for `raw_message_id`/`trader_id`, whitespace-only
  `symbol`/`action`, and non-`Decimal` prices.
- Unknown update field names and protected fields (`id`, `created_at`,
  `updated_at`) are rejected with `ValueError`.
- An empty update call is rejected with `ValueError`.
- Updating a missing id returns `None`.
- Repository writes (both create and update) are not committed
  automatically (verified with two connections to the same temporary
  database file).
- No row is written to `trade_signal_edits` on update, and no
  `INSERT INTO trade_signal_edits` SQL exists anywhere in
  `database/repository.py`.

**Result: 82/82 tests passing** (11 sources + 16 traders + 20 raw_messages +
35 trade_signals) via `python -m unittest discover -s tests -v`.
